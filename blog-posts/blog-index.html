Tech: SQL Injection

Saturday, September 20, 2014


    SQL injection is when SQL statements are inserted into a data field to be executed that are exploiting a security weakness and hoping to gain access to the database information. These vulnerabilities occur when situations such as the input data field not being filtered escape characters that exit string literals. This essentially allows a user (or in this case, a hacker) to access the database through data input fields. This means that someone can copy the database or even delete everything, or both! You can imagine that if a bank's website allowed this to happen, then someone could use SQL commands to access your information with out even having to login as you. That sounds pretty problematic, but luckily this is (hopefully) checked for and fixed.

    To avoid this, developers need to use input validation to check against pre-defined rules for the input type, such as length, type and syntax. You also want to make sure that any user input data is not sent to the SQL interpreter as a SQL query, as this allows commands to be sent directly through data fields. Input validation prevents commands from being passed this way, and limiting the user permissions as much as possible is always a good idea. When it comes to web applications, it is not recommended to use system administrator accounts, as this would have full, or much more significant privileges when it comes to the database access.

Tech: Class and Instance Methods

Saturday, September 13, 2014


    When comparing class methods and instance methods, the most basic difference is that instance methods are called on an instance of the class, while class methods are called on the class name itself. Instance methods, called on objects, have access the instance variables associated with the object. Because instance variables are associated with objects, our class methods can not access them because¸ after all¸ we aren't giving it any object to look at when calling the method.

    The benefits of instance methods are that we can access an objects information¸ and if we want¸ we can manipulate it to our liking. We need to have these methods because of the way objects are made. Objects all contain information, and that information applies only to that object. This is made convenient because of instance variables that are the variables that store the data associated with each object. We could do the same process without objects¸ but then we would have to create every variable for every object¸ which can be insanely cumbersome; imagine a database of a million people with all their personal information¸ and imagine how many variables you would need to hard code if we didn't have objects! So, since objects store this information so well¸ we need to have means of accessing and altering this information if necessary. This is where our instance methods come in. We may just want to ask the computer what one piece of data is, which means we would need to make a "getter" for that instance variable. To make sure we are referencing the specific variable value associated with the object, we have to call the getter method on the object itself, otherwise, how can it know what instance variable to return?

    Class methods are great for storing some sort of functionality that doesn't necessarily apply to specific objects. Say we want a method that takes all the instance variables of an object and interpolates them into a nicely formatted string. This could be done by passing an object to the method and returning the interpolated string with the objects variables in the string.

    Really you can make all methods class methods, but this is clunkier as you would have to pass the object in questions to each method and run the code on that object. But having instance methods allows us to skip the process of passing the object because by calling the method on that object, then it is implicit that that is the object you want to refer to.

    Here is a small example of a class called People¸ with an initialization method, which is our example of a class method, and a method #interpolate, which is an instance method. On line 14 I created an object called “me” that calls the initialization method of the People class, which is using the class method to create a new object. On line 15¸ I call me#interpolate¸ which calls the instance method #interpolate that takes all the data associated with "me" and prints/returns it as an interpolated string.

Tech: Ruby Classes

Saturday, September 6, 2014

    Object Oriented programming languages are all about objects and classes. Classes allow us to group similar objects together to minimize the work we need to do to create useful methods. For instance, if we did not have an Array class, with a list of a few dozen instance methods, then we would have to define each method every time we wanted to manipulate an array object we create. Because of classes, our Array class has that wonderful list of methods that can be applied to any object of that class. This is great because say we had a class called Library, where each object of the Library class was an individual book. We would not want to have to create a separate #check_out method for each and every book in the Library! That would be tedious and unnecessary. This is why we would create one single #check_out method in our Library class that will allow us to check out any book we want, by just calling that method on our object (remember, each book is an object).

    First, lets take a few steps back and talk about the creation of each object (book) in our Library class. Ruby allows us to create an #initialize method in our class, that is automatically called whenever we perform a Library#new call. This allows us to initialize a new object of the Library class.



    Here, line 9 creates a new object called book1 that is of the Library class, and it passes a title, author and ISBN to our initialize method. Again, this is just like if we were to say:

      book1 = Library.initialize("The Moon Is Down", "John Steinbeck", "1239755346").

    These pieces of information, or arguments, must be in the exact order you specify in the initialize declaration, otherwise you will end up with some funky titles, authors and ISBNs.

    Now that we have created a brand new object, book1, on line 1, we want to use a newly made Library#check_out method on it. Because book1 is an object or our Library class, we can use this #check_out method. This is because #check_out is an instance method, which means it can be used on instances of the class, and each object is an instance of a class, thus the instance method #check_out is usable on the instance of Library class book1.



    Lets take a look at the newly created instance method #check_out. This instance method has two parameters, the date and person. We have to pass this information to the method so we can know who checked out our book and when. Declaring this method within the class makes it an instance method for our Library class. What if we put it outside of the class you say? Well, then we could call #check_out on any object type we wanted, say, the crown jewels. It wouldn't make much sense to be able to check out the crown jewels now would it? That is why we declare the #check_out method in the class, we want it to be only usable on objects of our Library class and nothing else.

    So, now we have a method that lets us check out a book, but right now, we have no way of knowing if a book is checked out. It might be important to be able to know the status of @checked_out, which is why we have things called getters. Getters are convenient methods used for retrieving information on an object. Take a look:



    This method #checked_out? simply returns the value of @checked_out to us. Now when we enter the code on line 21, we get the current value of @checked_out. Getters typically come with setters, methods used for changing values associated with an object. We will want a getter for @checked_out because we will want to change that value depending on its status, however for values like the book title, we will not want setters since that is not going to change.



    Our method #check_in changes the values of @checked_out, @date and @person to reflect the now checked in status. We want to remove the values of date and person because we don't want their information to be reflected later on, and we need to also set @checked_out to false so when we call #checked_out? we will be given the correct value.

Tech: Hashes and Arrays

Friday, August 29, 2014



    Lately, we have been working a lot with hashes and arrays in Ruby. Hashes are very new to me, and determining which data structure is appropriate in what scenario takes some getting used to.

    An array can be as simple as a list of numbers, or it can be as complicated as lists of lists of lists of objects (that sounds overly complicated, right?). To reference a value in an array, you need to use that values index. Array indices start from 0, so the first item in an array will always have an index of 0, and the last element in an array will have an index of array.length – 1. The #length might look new to you, and if it does, it is a method that returns the length of whatever array you call it on. Arrays can be great for storing things like a list of words to be randomly selected for a game of hang-man, or the details of a person's address.

    Hashes are buddies with arrays, but they can be used very differently. For one, hashes are not bound to strictly numerical indices like arrays, in fact, you can give a hash any index you want. To avoid confusion, hashes don't really use index, they call it a "key." Every key in a hash is matched with a certain value, thus getting us key/value pairs. You might want to store a list of all of your friends and their ages, so you would assign the key as the persons name, and the value as their age. From here you can see a myriad of methods associated with hashes that allow you to play with the data, with things like sorting, deleting and adding methods.

    All I have learned so far in choosing is that it depends. I am even curious if a hash that has values of arrays, maybe that works, that could be pretty useful if you wanted to pull a list of numbers associated with a certain key. Luckily, there is quite a bit of overlap in methods, and they don't change naming conventions all over the place for each class, so you have a lot of repeated methods. Hashes are very cool, and I am glad Ruby has them, as arrays can sometimes be tricky to deal with, at least in the scenarios in which a hash is best.

Technical Blog: CSS Concepts Explained

Sunday, August 17, 2014


     For this weeks technical blog, the topic is CSS concepts and some differences between them.We will start with some basic concepts: Margin, Border, and Padding. These three things can be confusing to grasp at first as they are mostly just slightly different. I like to think about it like this; Imagine you are looking at a small town from a birds eye view, such that you can see all the buildings in the town. Those buildings represent elements on your website. The padding of an element is how much space is added inside the element. For example, the padding in our town example would be the space within a building that a person can roam. A small building without much space would be an element with only a little bit of padding, while large buildings would have lots of padding. Or you can think of it being the amount of yard/property a house has around before butting up against the next piece of property (element).

    Border is basically the outline of your elements. The house elements on our town website all have walls, and those walls outline the structure. Some structures like sheds might have thin walls (borders), while others might have thick ones.

    Margin is the amount of space outside of one elements border, that separates it from the next element. Imagine if people really disliked their neighbors, and could add fifty feet of no-mans-land space between their property and their neighbors. That would be adding a margin between their two elements.

    In CSS there are classes and ids. These handy tools allow you to be more specific about how you want certain elements styled, and to do it from your CSS document. Ids can only be used on one element on a page, and are designated with a # before the name of the id ( #idName).

    Classes can be applied to any number of elements, including elements with ids already. Elements can also have multiple classes each, you can have 5 elements with an id and 4 classes each assigned to it if you wanted, as long as it was only one id. Classes are denoted by a period at the start of the name, so: .className would be your syntax in your stylesheet. You can specify that classes only apply to certain elements by doing something like: div.className{}. Below you can see the class that is used to set up the table this blog is in. Each blog post is in its own table, therefore it is best to make this a class so that I can assign it to each blog post table.


    In positioning elements, you can use: absolute, fixed and relative. Absolute positions the element relative to its first positioned ancestor element. Fixed positions the element relative to the browser window. Relative positions the element relative to its normal position. So, a relative position with left:20 will add 20 pixels to the elements left position.

    The display property has two big values: inline and inline-block, as well as a long list of other possible values. Inline block displays the elements without starting a new line. So if you have a div that has a ul in it, if you do inline block, the list items will appear in a row. Using inline block allows you to form elements inline (such as with just using inline). The difference is that you get to treat it like a block in that you can still set width/height, margins and padding. This would cause a list of items to be in their own little boxes (blocks), while still being horizontally listed.

     Looking at Polygon from a distance, you are drawn to the articles, and pretty much every article has a corresponding image to help draw you in. The header of the site is a grid of the top/newest articles, featuring 4 or 5 different options. These options aren’t just your basic text link, they are awesome and curious photos with the article title overlaid. These links are great because they give you more to base your choice to click on other than just the wording of the title.

     The navigation bar allows you to quickly and efficiently narrow down the topics you want covered. You can even choose what gaming system you want to focus on, or video reviews of games of all types. The nav bar remains in place no matter where on the site you are, and to make it easier, Polygon will suggest related articles at the bottom of articles you read.

5 adjectives about the website visually: organized, attractive, themed, divided, flowing

5 adjectives to describe content, focus and purpose: gaming, broad-spectrum, informative, dedicated, well-stocked.

Some adjectives that I feel after being on the site for a while: bored (Well, I can only read about gaming news for so long), anxious (new games coming out!), entertained.

Polygon is also a branch of The Verge which is the more generalized tech world news site, which has many of the same qualities as Polygon does.

Next we have Wall Street Journal:

     The Wall Street Journal does a good job of incorporating technology into the timeless newspaper style layout. At a glance your eyes are drawn to the most prominent articles, the front pagers. The most prominent feature on the site is the WSJ logo, it seems to pop out too much in its box in the top left corner. It would look better with a centered and written out title, as opposed to the current large text “WSJ.”

     WSJ takes a large amount of content in the form of news articles and links, and organizes it into categories. Featured content gets the more prominent locations and visuals, while other content is relegated to text links on the margins. Their layout allows the less prominent content to still be quite visible and accessible.

     For a site that really is loaded with content, and is re-loaded every day, the navigation lends itself to easy use. The navigation bar has a well thought out selection of topics to choose from to narrow your search for interesting articles.

5 adjectives to describe the site visually: loaded, organized, compartmentalized, relevant, intuitive

5 adjectives to describe content, focus and purpose: informative, detailed, comprehensive, current, dedicated.

Some adjectives describing how I feel after some time on the site: educated, informed, overwhelmed.

My final choice is going to have to be IMDB:

     IMDb is a website dedicated to information on television entertainment. This site gives a wealth of information for each and every move, tv show, or actor/actress I have ever looked up, and proves to be quite comprehensive. If you ever have one of those “Oh who was that guy in that movie that one time?” moments, IMDb is the place to look.

     The homepage offers interesting facts and videos/graphics of recent or older media. I tend to not go straight to the homepage as I usually get to this site through a Google search of a movie or actor. What really is the essential part of the home page is the search bar, and it is the first thing your eye catches. The search function is very effective, and will yield specific results. From farther back, your eye tends to be moved to the different videos and images screaming for attention lower down on the page.

5 adjectives to describe the site visually: media rich, focused, informative, alluring, inquisitive.

5 adjectives to describe content, focus and purpose: specific, thorough, related, extensive, complete.

Some adjectives to describe how I feel after some time on the site: satisfied (Gotta love getting that movie trivia factoid answered), entertained.
